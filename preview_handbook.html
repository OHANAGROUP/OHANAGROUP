01 · Engineering Doctrine

La simplicidad operativa es una feature

Cada componente de infraestructura introduce costo de mantenimiento, overhead cognitivo, riesgo operacional, propagación de fallos y complejidad de debugging. Por eso la infraestructura debe justificar activamente su existencia.

Local-first
donde sea factible
Observable
desde el día uno
Cost-aware
en cada loop
Failure-tolerant
bajo estrés extremo
Small teams
recursos acotados

Priorizamos comportamiento operacional determinista sobre escalabilidad distribuida prematura. Las restricciones son tratadas como inputs arquitectónicos de primera clase, no como inconveniencias temporales.

03 · Systems Engineering Standards

Presupuestos operacionales & de complejidad

A

Operational Budgets

  • ›Execution Ceiling: ≤ 4 vCPUs y ≤ 16 GB RAM por servicio.
  • ›Inferencia: Ollama CPU-bound para tareas estándar, cloud solo para transacciones de alta confianza.
  • ›Cloud runtimes 24/7: huella de costo cercana a cero.
B

Complejidad Prohibida

✕ Kubernetes ✕ Consul / Etcd ✕ Kafka ✕ microservicios innecesarios ✕ vector DBs externos

Solo se permiten si métricas objetivas demuestran necesidad absoluta.

C

Observabilidad

/healthrequired
JSON structured logsrequired
Prometheus metricsrequired
Queue depth & throughputrequired
Error countersrequired
D

Escalation Philosophy

  1. 1Local recovery — auto-healing vía engine reset.
  2. 2Retry w/ backoff — exponential delay stateful.
  3. 3Graceful degradation — payloads cacheados.
  4. 4Operator alert — solo tras agotar todo lo anterior.
⚠ Prohibidos los retries automáticos en cascada entre providers (evita bill shock).

06 · Failure Modes Mapping

Mapa de fallos y mitigaciones

Failure Mode Impacto Mitigación Técnica
SQLite WAL
Write latency spikes
 Spikes de latencia en escrituras paralelas. Mutex locks serializados en el módulo core/db_sync.
Ollama RAM thrash
Inference freeze
 El motor de inferencia se congela al cambiar modelos en CPU. Reglas estrictas de keep_alive + thread allocation vía watchdog.
Network starvation
Cloud sync fail
 Caídas WAN abortan ciclos de sync a la nube. Offline queuing en SQLite, defer Supabase hasta ping ok.
Playwright leaks
RAM exhaustion
 Browsers de automatización consumen RAM sin liberar. Reciclaje estricto de procesos + triggers de memory cleanup en workers.
07 · Architectural Tradeoffs

Decisiones conscientes, costos aceptados

✓ Beneficios

  • ▸Resiliencia offline total
  • ▸Baja latencia (archivo local)
  • ▸Simplicidad: un archivo, cero servicios

✕ Consecuencias aceptadas

  • ▸Mayor write contention vs Postgres
  • ▸Requiere mutex locks bien diseñados
  • ▸No escala horizontalmente

✓ Beneficios

  • ▸Facturación de inferencia = $0
  • ▸Privacidad de datos absoluta
  • ▸Independencia del edge

✕ Consecuencias aceptadas

  • ▸Menor throughput (tokens/s) vs GPU cloud
  • ▸Modelos más pequeños (Q4/Q5)
  • ▸Latencia mayor en prompts largos

✓ Beneficios

  • ▸Cambios rápidos en codebase
  • ▸Cero latencia entre servicios
  • ▸Bajo overhead mental

✕ Consecuencias aceptadas

  • ▸Granularidad reducida de escalado
  • ▸Deploy atómico de todo el sistema

✓ Beneficios

  • ▸Containerización estándar
  • ▸Recuperación rápida de entorno
  • ▸Cero vendor lock-in

✕ Consecuencias aceptadas

  • ▸Single-node por diseño
08 · Explicit Non-Goals

Lo que NO construimos

🚫

Kubernetes

Usamos la simplicidad de Docker Compose single-node.

🚫

Clusters de inferencia distribuida

No escalamos modelos entre múltiples GPUs en red.

🚫

GPU scheduling de alta frecuencia

Optimizamos threads CPU para hardware low-power.

🚫

Colas globales distribuidas

Toda secuenciación de tareas es local-first.

🚫

Streaming pesado (Kafka)

Archivos locales y colas DB lightweight.

🚫

Active-active multi-región

Sync asíncrono a un único nodo cloud.

09 · Architecture Decision Ledger

ADRs bajo /docs/adr/

Todo cambio estructural se documenta con el modelo estricto de 5 bloques.

ADR-001 FastAPI + SQLite WAL + Ollama como core stack
1 · Context

Hardware limitado a 4 vCPUs/16 GB, equipo de un solo operador, presupuesto cloud ~$0, necesidad de inferencia privada.

2 · Decision

FastAPI como gateway async, SQLite en WAL mode como cola local, Ollama CPU-bound con Gemma4 como modelo de trabajo.

3 · Tradeoffs

+ Cero cloud bill, + simplicidad extrema. − Throughput limitado en escrituras concurrentes, − modelos más pequeños.

4 · Consequences

Requiere módulo core/mutex, tests de contención, monitoreo de queue depth.

5 · Rejected Alternatives

Postgres+Redis (overkill, costo cloud), Kafka (complejidad injustificada), Kubernetes (contradice single-operator).

ADR-003 Supabase como único nodo cloud async draft
ADR-004 Playwright con reciclaje estricto de procesos approved
